01 · Programme

Platform Overview

Project Nexus is a unified digital portal replacing BT Wholesale's fragmented legacy estate with a single self-serve portal for all partner tiers. The core RBAC model (permissions, roles, user management) is built once and shared across the entire portal.

Legacy portals being replaced

VEL / Business PortalEnd-user and in-life management (SSO'd from BZ) — out of scope for Nexus redesign; separate team
Hub / Partner PlusBroadband One, WHC Express, Complete Switch; current user management model
SAFService Assurance / Fault Diagnostics

Success metrics (PRD)

  • NPS: +12 → +33 (+20 points)
  • 90% strategic product adoption within 24 months of launch
  • 40% faster time-to-market for new products
  • 20% cost-to-serve reduction · 99.9% portal and API uptime
API-first principle (PRD verbatim)

"No business capability exists in the UI without it being driven by an API. Any API could be externalised at any time. We will use the same APIs as our customers."

02 · Programme

Organisation Hierarchy

BT Wholesale only has a direct commercial relationship with Resellers (Direct/Super). All other tiers operate under their parent's contract.

TierDescriptionKey constraint
ResellerDirect contract with BT Wholesale. Full downstream control.None — max privilege
Sub-ResellerNo direct BT contract. Sells under parent's relationship.No support tickets; no billing; no BZ access
Child ResellerBroad downstream access with selected restrictions.Cannot raise support tickets — must go via parent
DealerMost restricted. Inherits parent's branding only.No user management; limited ordering; no portal login (TBC)
Design principle

Build for Direct Reseller at max privilege first, then subtract downward per tier. Never design for the most restricted first.

Example org tree (used in prototype)

  • Northgate Telecom (Reseller) — Metro Connect (Sub), Halo Networks (Child), Riverside Comms (Sub), Apex Telecom (Dealer)
  • Beacon Wholesale Ltd (Reseller)

09 · Context

PRD — Key Decisions

Source: Nexus Product Requirements Document (65pp, draft — no version/date stamped). Last read into intel: 23 Jul 2026.

Resolved 21 Jul 2026

Identity platform: Salesforce RBAC confirmed. Strategic Identity ruled out (too slow). PRD explicitly flags: "Wholesale identity is NOT finalised" — this was the open blocker; Salesforce decision closes it pending Vanmathi confirmation.

Resolved — PRD defines

RBAC model: 3-layer — (1) Permission Sets from a granular catalogue grouped by capability, (2) Roles as named combinations of permission sets, (3) Users assigned one or more roles. Role changes take effect immediately. All changes audit-logged.

RBAC enforcement rules (PRD)

  • Enforced consistently across portal UI, APIs, and backend services
  • BT internal users acting on behalf of a partner cannot bypass partner permissions
  • All role/permission changes audit-logged: who, what, when, outcome
  • Unauthorised actions blocked with a clear message — no silent failure

Hierarchy rules (PRD)

  • Sub-reseller inherits constraints from parent — cannot exceed parent permissions
  • Products/journeys enabled for sub-reseller must be ≤ parent reseller's capabilities
  • Constraint enforced across UI, APIs, and backend validation — not UI-only
  • BT Internal users cannot register or onboard new organisations — only manage existing
Blocker (PRD, open)

PRD explicitly states: "Wholesale identity is NOT finalised — This impacts access, integrations, and user experience across everything." Salesforce decision (21 Jul) addresses the platform choice; detailed integration design is still outstanding.

User onboarding flow (PRD)

  • BT Admin creates organisation → downstream systems auto-provisioned (Identity, CUG, billing references)
  • Admin sends invitation email with registration link — expiry and resend supported
  • If user already exists in identity system, access is linked — no duplicate identity created
  • When access revoked: active sessions terminated AND API access revoked immediately
Open (PRD)

"Do we need audit logs and access to them (screens or backend logs)?" — not yet resolved. NFRs confirm audit trail is required; the design question is whether it's surfaced as a UI screen.

10 · Context

Design Decisions & Constraints

  1. Design for Direct Reseller first — North Star at max privilege, subtract downward for Child/Sub/Dealer.
  2. Child Reseller = Direct Reseller minus support tickets — this is the primary differentiator between the two tiers.
  3. PRD is source of truth — Mural notes and session recordings may be stale or pre-product.
  4. Customer establishment is out of scope for the strategic portal (Service Designer territory).
  5. VEL redesign is a separate team — Nexus enables SSO access to VEL but does not replace it.
  6. Sub-Reseller Admin billing exclusion is an RBAC enforcement requirement, not a UI-only hide.
Open design decision

Are roles BT-fixed or reseller-configurable? The prototype currently treats roles as a fixed set. This is a key product question that affects the entire RBAC model — resolution needed before detailed design.

Open design decision

Does Dealer need portal login at all, or are they just an entitlement recipient? May simplify the hierarchy significantly if Dealer is off-portal.
