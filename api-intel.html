Who has access

API access is for direct Partners only. developer.bt.com is only available to organisations with a direct contractual relationship with BT Wholesale. Partner Organisations (child resellers) do not have API access — their relationship is with the Partner, not BT.

This is a significant constraint for Nexus design: any API-related features on the portal (API keys, sandbox apps, usage dashboards) are only relevant to the Partner tier, not Partner Organisations.

The portal is also shared across BT Group — not wholesale-exclusive. Other teams on developer.bt.com include Active Intelligence (location data APIs) and an Identity team.

developer.bt.com

The developer portal is publicly accessible (pre-login) and provides documentation, tutorials, and code examples. Logging in adds a Dashboard for managing sandbox and production apps.

Pre-login

  • API catalogue, tutorials, process diagrams, code examples
  • Wholesale APIs buried — not on homepage, must navigate to API Products → Wholesale
  • Documentation is very detailed — the simplest API (Address Management) has a full tutorial; the Order API has introduction, prerequisites, provide, modify, tables of codes, response examples, and process diagrams

Post-login

  • Dashboard appears — shows sandbox and production apps
  • Can create sandbox apps instantly (no approval needed)
  • Production apps require BT admin approval (email-based workflow)
"I haven't even logged in. So I'll do the next thing and log in just to show what additional thing that gives you."
— John Daffern
Discovery gap. A new partner arriving at the homepage would find nothing about wholesale broadband or any wholesale products. The discoverability of wholesale APIs on the shared portal is poor.

API journey (broadband example)

The API journey mirrors a portal order flow — the same steps apply whether via API or UI. Each step depends on the previous one.

1
Address lookup — provide postcode (+ optional house number) → receive Openreach UPRN (unique address code). Also supports UPRN-direct lookup.
2
Products & network availability — what speeds/products are available at that address, whether there's an existing line in use.
3
Appointments — check engineer availability. Some installs don't require an appointment (e.g. working line present).
4
Product selection & order submission — submit the order with chosen product, appointment, and address details.

Customers are expected to understand the sequencing themselves with TMF APIs — this is a large part of the complexity burden Matthew's Experience API is designed to eliminate.

Authentication & keys

  • OAuth — standard authentication flow. Apps use a Key + Secret (like username + password) to obtain a token.
  • Token required before any API call can be made.
  • Sandbox apps — Key + Secret available immediately on creation. No approval needed.
  • Production apps — no Key + Secret until BT admin approves. Production keys must be associated with a verified company to prevent fraudulent orders.

Sandbox vs production

Sandbox
  • Instant access — no approval
  • No risk — can't break anything
  • No account manager involved
  • Endpoint: api-sandbox.wholesale.bt.com
  • Same request/response format as production
Production
  • Requires BT admin approval (email workflow)
  • Keys linked to verified company
  • Endpoint: api.wholesale.bt.com
  • Code is identical — only hostname changes
  • Live orders, live addresses
Gap: no migration path from sandbox → production. Currently "contact your account manager" is the answer. There's no proper self-service or guided process to move from "I've tested my code and it works" to "I'm now live." John confirmed this is a known gap with no current solution.

TMF APIs — current state

TMF (TM Forum) APIs are the telecom industry standard. They are low-level, powerful, and flexible — but put the full complexity burden on the customer's development team.

"The expectation is that we get these developers to make lots of calls, all of these different services… a modify is different from an amend. Yet to most people that's literally the same thing. It is not in the world of telco."
— Matthew Lindley

What customers must handle themselves with TMF

  • Know the correct API sequencing (address → availability → appointment → order)
  • Understand Openreach codes and what to do with each response
  • Handle all business rules (what's a modify vs an amend vs a cease)
  • Build orchestration logic to stitch calls together
  • Interpret complex, multi-field JSON responses
  • Know telecom-specific terminology (GEA, FTTP, UPRN, OMT, ORDI…)

Who can afford to do this

  • Large enterprise partners (Gamma, Giacom, Virgin Media)
  • "API-first" companies — IT specialists for whom this is their core competency
  • Both groups invest heavily: architects, solution designers, developers, months of work
High barrier to entry. Standing up a TMF integration can take 12–18 months and is a major capital expense. Only the largest partners tackle it. Smaller partners are effectively locked out of programmatic access.

Experience APIs — Matthew Lindley's workstream

Matthew is building an orchestration layer — the Experience API — that sits in front of the TMF APIs and absorbs complexity on behalf of the customer.

Analogy. TMF = raw building materials (you coordinate builders, electricians, plumbers yourself). Experience API = a project manager who handles all the specialists and only asks you for what they genuinely need from you.
TMF API — customer's responsibility
  • Call address API
  • Call availability API
  • Call appointment API
  • Understand all responses
  • Apply business rules
  • Handle errors and edge cases
  • Submit order API
Experience API — BT's responsibility
  • Customer provides: address
  • BT runs: lookup, availability, rules
  • Customer receives: available products
  • Customer selects: product + appointment
  • BT handles: order orchestration
  • Customer receives: simple confirmation

Hypermedia / "what's next"

A core design principle: every API response contains links to valid next actions. Customers don't need documentation open beside them — the response tells them what they can do next.

"This is your next step."
— Matthew Lindley, on the Experience API design intent

Invalid actions are omitted from responses rather than returned as errors. Example: once an order passes the point of no return, the cancel link simply disappears, and the API explains why.

AI compatibility

Matthew specifically designed responses to be human-readable and AI-consumable. Tools like Copilot, Gemini, and Claude can parse the API definitions and generate forms, portals, and integration code from them. John demonstrated this with a Google Sheets demo — 800+ lines of JavaScript, 100% AI-generated via Google Anti-Gravity / Gemini.

Key shift. TMF: "Tell me exactly what you want and how to do it." Experience API: "Tell me your goal and I'll guide you there." This is the philosophical difference — and it has direct implications for what the Nexus portal needs to expose to Partners.

Error handling model

TMF (current)

Customer handles all errors. System returns telecom-specific error codes; customer's application must interpret them and decide what to do next.

Experience API (proposed)

Errors are absorbed or translated by BT's orchestration layer. Instead of raw error codes, the API returns business-friendly messages and always includes valid next actions.

ScenarioTMF responseExperience API response
Order can be cancelledCustomer checks status codes and conditionally shows CancelCancel link present in response
Order cannot be cancelledCancel API returns error; customer handles itCancel link absent; reason provided in response
Wrong API sequenceError returned; customer must diagnoseExperience API enforces sequence; next step always explicit
Telecom code returnedCustomer must look up code meaningHuman-readable message returned

The principle: prevent errors rather than handle them after the fact. The front end shows only what the API says is valid.

UX observations & design implications

API in the portal ≠ API integration. The portal and the API are separate channels. Most large partners never use the portal at all. When they do, it's for exception handling — not routine ordering. Any API features in Nexus (key management, usage, sandbox) serve a small but important subset of Partners.

What Nexus may need for the API workstream

  • UX  API key management — Partners need to create, view and rotate sandbox and production keys from within Nexus
  • UX  Sandbox → production onboarding — there's currently no UX for this transition. A guided flow in Nexus could fill the gap
  • UX  Usage / analytics dashboard — currently no meaningful reporting exists. If BT resolves the Apigee → GCP export, Nexus is the natural home for this
  • UX  API access is Partner-only — Partner Organisations should never see API features; role and org-type gating required
  • UX  API persona in RBAC — Matthew Lindley's workstream (mentioned separately in RBAC intel) maps to this; need to understand the API Developer role

Experience API implications for portal design

  • If the portal consumes Experience APIs rather than TMF, the ordering and management flows become much simpler to build
  • Show/hide controls can be driven by what the API returns — no complex front-end state logic
  • The portal becomes a thin UI layer over the Experience API; the same API could serve both the portal and partners' own CRM integrations
