Summary

The gap analysis ran on 24 Jul 2026 against the RBAC PRD (Section 17.5, Section 17.6, Section 17.9, Section 17.10), the RBAC-intel.md doc, and the wholesale.html prototype. It identified 11 things that could be built now, 6 blocked on open questions, and 4 areas the prototype doesn't model at all.
All 11 actionable gaps were implemented in a single session on 24 Jul 2026. The prototype is now current with everything that can be built given the open questions outstanding.
The prototype is a single-file React UMD prototype at wholesale.html. It's a design and stakeholder tool — not production code. Decisions made here inform the Salesforce RBAC implementation.

Blind spots

Areas the prototype doesn't model at all — not blocked, just not yet in scope. Worth flagging before detailed design.
B1
API consumer RBAC scopes
The prototype has an API Developer role and API key management UI, but doesn't model how API keys map to permission sets or what scopes an API consumer gets. This is the intersection with Matthew Lindley's API workstream.
Blind spot
B2
Multi-tenant data isolation
The prototype shows who can see what in the UI but doesn't model or surface how data leaks between orgs are prevented at the API/data layer. A reseller admin theoretically can't see another reseller's data — but this isn't shown or enforced in the prototype.
Blind spot
B3
Role inheritance across org tiers
PRD Section 17.5 mentions a three-layer model but is silent on whether permissions cascade down the org tree. If a Reseller Admin grants a role, does that role also apply within their child orgs? The prototype doesn't model this.
PRD Section 17.5
Blind spot
B4
Offboarding / removal cascade
What happens to open orders, active faults, and billing queries when a user is permanently removed? The prototype treats removal as a user-only action but in practice it may need to reassign or close linked records. No design exists for this yet.
Blind spot

Open questions

Questions that need answers before the blocked items can move. Sourced from RBAC-intel.md Section 12 and the gap analysis.
QuestionWhy it mattersWho to ask
Can resellers create their own custom roles, or are roles BT-fixed? Determines whether the "Create custom role" feature stays in the prototype or gets removed. Currently built but gated behind an info toast. Neil
What is the exact invitation expiry window? Expired chip exists but the prototype doesn't show a countdown or expiry date. PRD is silent on the window (24h? 7 days?) PRD / Neil
Do Salesforce permission sets map 1:1 to the RBAC permission sets in the PRD? If yes, the permission set catalogue can be built directly. If not, there's a translation layer to design. Vanmathi
What are the SLAs for primary contact changes? The contactChangeWiz says "confirmation email before change takes effect" but the actual SLA (instant? 24h? requires BT approval?) is unknown. PRD / Neil
Does remove require BT approval or is it self-service? Prototype now says "contact BT Wholesale support" to reverse — but the actual removal action is self-service. Confirm this split is correct. Neil / Service Owner
What data does a sub-reseller admin see from their parent? Sub-reseller data visibility UI is built with three toggles, but the actual default state (open / closed) and what BT controls vs reseller controls isn't defined. Neil / PRD Section 17.2
